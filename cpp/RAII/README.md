# RAII

❓ Что такое `RAII`? Привести примеры из `STL`.

- а если исключение?
- а если ресурс не память?
- где `RAII` ломается?

## Senior-like ответ

`RAII` (`Resource Acquisition Is Initialization`) - это идиома C++, в которой владение и освобождение ресурса привязаны к `lifetime` объекта: ресурс захватывается в конструкторе, освобождается в деструкторе, а корректный `cleanup` обеспечивается типом и `scope`, а не ручной дисциплиной. Это дает `deterministic cleanup`, упрощает `ownership` и делает код `exception-safe` в части освобождения уже захваченных ресурсов.

Хороший RAII-тип держит простой инвариант владения, имеет корректный `moved-from state`, а его деструктор не бросает исключения. RAII не заменяет продуманную модель `ownership`, но делает освобождение ресурсов автоматическим, локальным и безопасным.

Примеры из `STL`:

- `std::unique_ptr` - освобождает память;
- `std::shared_ptr` - освобождает ресурс при уходе последнего владельца;
- `std::vector` / `std::string` - управляют своей внутренней памятью;
- `std::lock_guard`, `std::unique_lock`, `std::scoped_lock` - захватывают и освобождают `mutex`;
- `std::fstream` - закрывает файл;
- `std::jthread` - связывает `thread lifetime` с объектом.

А если исключение?

- Во время `stack unwinding` будут вызваны деструкторы уже полностью сконструированных объектов, поэтому RAII и дает `exception-safe cleanup`.

А если ресурс не память?

- RAII одинаково применим к `mutex`, `fd`, `socket`, `FILE*`, `transaction`, `mapped memory`, временным изменениям состояния и любому другому ресурсу с парой `acquire/release`.

Где RAII ломается?

- когда освобождение ресурса нельзя надежно выполнить в `noexcept dtor`;
- когда ресурс живет не по `scope`, а по внешнему протоколу или распределенному жизненному циклу;
- когда ownership не выражен явно и один ресурс считают своим несколько объектов;
- когда нужен явный `commit` / `close` с обработкой ошибки, а `dtor` может быть только аварийной страховкой.

## Что это значит на практике

- объект в `ctor` принимает владение некоторым ресурсом;
- после успешного завершения `ctor` соблюдается инвариант:
  - объект либо валидно владеет ресурсом;
  - либо находится в корректном состоянии "ресурс отсутствует";
- `dtor` освобождает ресурс автоматически;
- управление ресурсами становится детерминированным и привязанным к `lifetime` объекта.

RAII - это способ сделать правильное поведение дефолтным поведением типа.

## Что может быть ресурсом

В контексте RAII ресурсом может быть почти все, что требует явного `acquire/release`:

- динамическая память;
- `mutex` / `spinlock` / `rwlock`;
- `file descriptor`;
- `socket`;
- `FILE*`;
- `database connection`;
- `transaction` / `rollback guard`;
- `mapped memory region`;
- временное изменение `global` / `thread-local state`;
- `handle` на объект ОС;
- `registration/unregistration` в callback-системе;
- `open/close` для некоторого subsystem object.

RAII - это про обобщенное безопасное владение ресурсом.

## Зачем это нужно

Без RAII код управления ресурсом обычно выглядит так:

```cpp
resource_acquire();

do_work();

resource_release();
```

Проблемы такого подхода:

- легко забыть `release`;
- легко сделать ранний `return` и не освободить ресурс;
- легко выбросить исключение между `acquire` и `release`;
- сложно поддерживать много веток выполнения;
- код с несколькими ресурсами становится хрупким и трудночитаемым.

RAII решает это так:

```cpp
ResourceGuard guard(...);
do_work();
```

Освобождение происходит автоматически при выходе из `scope`:

- по обычному завершению;
- по `return`;
- по `break` / `continue`;
- по исключению.

Это дает:

- `deterministic cleanup`;
- `exception safety`;
- меньше ручного `bookkeeping`;
- проще `reasoning about ownership`;
- проще поддержка инвариантов.

## На чем держится RAII в C++

RAII опирается на фундаментальные свойства языка C++:

- автоматические объекты имеют детерминированный `lifetime`;
- деструкторы вызываются автоматически;
- во время исключений происходит `stack unwinding`;
- в языке есть конструкторы, деструкторы и строгая модель инициализации.

Что это означает:

- локальная переменная в блоке уничтожается при выходе из блока;
- деструкторы вызываются для автоматических объектов, временных объектов, членов класса, базовых классов и так далее;
- при раскрутке стека вызываются деструкторы уже сконструированных объектов;
- `acquire/release` можно оформить как часть самого типа.

## Порядок разрушения важен

Для RAII важно понимать, в каком порядке уничтожаются объекты:

- локальные объекты в одном `scope` уничтожаются в обратном порядке создания;
- члены класса уничтожаются в обратном порядке объявления в классе;
- при разрушении объекта сначала выполняется тело его деструктора, затем уничтожаются поля, затем базовые классы.

Это важно, потому что один ресурс может зависеть от другого.

```cpp
struct X {
    Logger log;
    File file;
};
```

Здесь сначала уничтожится `file`, потом `log`, потому что члены разрушаются в обратном порядке объявления.

Если зависимости между ресурсами есть, порядок полей в классе должен отражать этот контракт.

## RAII и исключения

При выбросе исключения начинается `stack unwinding`, например:

```cpp
void f() {
    ResourceGuard g1;
    ResourceGuard g2;
    dangerous_operation(); // бросает исключение
}
```

Если `dangerous_operation()` бросает исключение, то:

- сначала уничтожается `g2`;
- потом `g1`;
- освобождаются их ресурсы;
- управление передается ближайшему подходящему `catch`.

RAII обеспечивает `exception-safe cleanup`.

Без RAII пришлось бы вручную писать:

```cpp
acquire1();
acquire2();

try {
    dangerous_operation();
} catch (...) {
    release2();
    release1();
    throw;
}

release2();
release1();
```

## Важное правило про деструкторы

Деструктор не должен выбрасывать исключения.

Если `dtor` выбросит исключение во время `stack unwinding`, а в этот момент уже летит другое исключение, программа завершится через `std::terminate`.

Практическое правило:

- деструкторы RAII-типов должны быть `noexcept` по смыслу.

```cpp
~ResourceGuard() noexcept {
    release_noexcept();
}
```

Если операция освобождения может завершиться ошибкой, есть два типовых варианта:

- скрыть ошибку внутри `dtor` и залогировать ее;
- дать пользователю отдельный явный метод вроде `close()` / `commit()`, который можно вызвать до разрушения объекта и обработать ошибку явно.

`dtor` в таком дизайне остается только аварийной страховкой.

## Что если конструктор не завершился

Если `ctor` объекта не завершился успешно, `dtor` самого объекта не вызывается, потому что объект считается не полностью сконструированным.

Но при этом:

- `dtors` уже успешно сконструированных членов будут вызваны;
- `dtors` базовых классов, которые успели сконструироваться, тоже будут вызваны.

```cpp
class X {
    MemberA a;
    MemberB b;

public:
    X() : a(...), b(...) {
        // если здесь будет exception,
        // dtors a и b вызовутся автоматически
    }
};
```

RAII-класс должен поддерживать сильный инвариант владения.

## Инвариант RAII-типа

У хорошего RAII-типа обычно есть очень простой инвариант:

- либо объект владеет валидным ресурсом;
- либо объект находится в пустом состоянии и ничего не должен освобождать.

Пустое состояние нужно как минимум для:

- `move`-операций;
- `default construction`, если он вообще разрешен;
- `release()` / `reset()`-подобных операций.

## Пример 1: `mutex guard`

```cpp
void f(std::mutex& m) {
    std::lock_guard<std::mutex> lock(m);
    critical_section();
}
```

Что тут происходит:

- в конструкторе `lock_guard` вызывает `m.lock()`;
- в деструкторе вызывает `m.unlock()`;
- даже если `critical_section()` бросит исключение, `mutex` будет освобожден.

## Пример 2: управление памятью

Плохой вариант:

```cpp
void f() {
    int* p = new int(42);
    g(); // может бросить
    delete p;
}
```

Если `g()` бросает, `delete p` не выполняется, значит будет `leak`.

RAII-вариант:

```cpp
void f() {
    std::unique_ptr<int> p = std::make_unique<int>(42);
    g(); // может бросить
}
```

Теперь память освобождается автоматически в деструкторе `unique_ptr`.

`STL`-контейнеры и `smart pointers` - фундаментальные RAII-типы.

## Пример 3: `file handle wrapper`

```cpp
class File {
public:
    explicit File(const char* path) {
        fd_ = ::open(path, O_RDONLY);
        if (fd_ == -1) {
            throw std::runtime_error("open failed");
        }
    }

    ~File() noexcept {
        if (fd_ != -1) {
            ::close(fd_);
        }
    }

    File(const File&) = delete;
    File& operator=(const File&) = delete;

    File(File&& other) noexcept : fd_(other.fd_) {
        other.fd_ = -1;
    }

    File& operator=(File&& other) noexcept {
        if (this != &other) {
            if (fd_ != -1) {
                ::close(fd_);
            }
            fd_ = other.fd_;
            other.fd_ = -1;
        }
        return *this;
    }

    int get() const noexcept { return fd_; }

private:
    int fd_{-1};
};
```

Что тут важно:

- `fd_` - это `ownership-bearing state`;
- конструктор либо открывает файл, либо бросает исключение;
- деструктор закрывает ресурс;
- копирование запрещено, потому что дескриптор имеет уникальное владение;
- перемещение передает `ownership`.

Практическая тонкость: `::close()` тоже может вернуть ошибку, но бросать из деструктора нельзя. Поэтому для реального wrapper-а часто делают так:

- `close()` - явный метод, который возвращает статус;
- `~File()` - best-effort cleanup без исключений.

## RAII и ownership semantics

RAII почти всегда связан с вопросом: кто владеет ресурсом.

### 1. Unique ownership

Один объект владеет ресурсом.

Примеры:

- `std::unique_ptr`;
- `std::lock_guard`;
- `file handle wrapper`.

### 2. Shared ownership

Ресурс освобождается, когда уходит последний владелец.

Пример:

- `std::shared_ptr`.

Это тоже RAII, но уже через `shared lifetime`.

### 3. Non-owning handle/view

Объект не освобождает ресурс, а только ссылается на него.

Примеры:

- `raw pointer` как `observer`;
- `std::string_view`;
- `iterator`;
- `span-like object`.

Такие типы не являются `owner`-ами и потому не реализуют RAII-освобождение ресурса.

## RAII и move semantics

Современный RAII в C++ тесно связан с `move semantics`.

Если объект владеет уникальным ресурсом, копирование часто нельзя разрешить, но можно разрешить перемещение.

`moved-from state` должен быть безопасным для destruction.

Частое правило проектирования:

- если тип владеет уникальным low-level ресурсом, он обычно `non-copyable` и `movable`;
- если тип просто агрегирует другие RAII-типы, часто лучше опираться на `Rule of Zero`.

## Rule of Three / Five / Zero

RAII тесно связан с семейством правил:

### Rule of Three

Если типу нужен пользовательский `destructor`, `copy constructor` или `copy assignment`, почти наверняка нужны все три.

### Rule of Five

В современном C++ к ним добавляются еще:

- `move constructor`;
- `move assignment`.

Если тип вручную управляет ресурсом, обычно нужно явно подумать обо всех пяти специальных функциях, даже если часть из них будет `= delete`.

### Rule of Zero

Лучший случай - не писать специальные функции вообще, а собирать тип из готовых RAII-компонентов:

```cpp
struct Session {
    std::string name;
    std::unique_ptr<Connection> conn;
    std::vector<std::byte> buffer;
};
```

Если владение уже корректно выражено через поля, компилятор и стандартные типы делают большую часть работы сами.

## RAII и стандартная библиотека

Стандартная библиотека C++ массово использует RAII.

Самые характерные примеры:

- `std::unique_ptr` - управление уникальным владением памятью;
- `std::shared_ptr` - `shared ownership`;
- `std::lock_guard` / `std::unique_lock` / `std::scoped_lock` - управление `lock lifetime`;
- `std::fstream` - управление файловым ресурсом;
- контейнеры `STL` (`vector`, `string`, `map` и так далее) - управление внутренней динамической памятью и `lifetime` элементов;
- `std::jthread` - `join/stop semantics` с RAII-характером управления потоком.

Важно: контейнеры управляют своими внутренними ресурсами через RAII, но выделение памяти не обязательно происходит только в конструкторе; оно может происходить по мере роста контейнера.

### `std::lock_guard` vs `std::unique_lock`

`std::lock_guard`:

- минималистичный guard;
- всегда владеет `lock`-ом;
- не умеет `unlock()` / `lock()` вручную;
- почти без накладных расходов.

`std::unique_lock`:

- более гибкий;
- может быть создан с `defer_lock`;
- умеет передавать владение `lock`-ом;
- нужен для `condition_variable`.

Если нужна только простая защита критической секции, обычно достаточно `lock_guard`.

## `release()` и передача владения

Для некоторых RAII-типов полезна операция передачи владения наружу.

Пример с `std::unique_ptr`:

```cpp
std::unique_ptr<T> p = std::make_unique<T>();
T* raw = p.release();
```

После `release()`:

- RAII-объект больше не владеет ресурсом;
- ответственность за `release` ресурса переходит вызывающему коду.

Это опасная операция, потому что она временно выводит ресурс из-под RAII-контроля.

Предпочтительно держать такие переходы редкими и явно мотивированными.

## RAII и commit/rollback-паттерн

RAII полезен не только для "простого освобождения", но и для временных изменений состояния.

Типичный паттерн:

- в конструкторе фиксируется старое состояние;
- в деструкторе состояние откатывается;
- при успехе вызывается `commit()` / `dismiss()`, после чего откат не нужен.

Это удобно для:

- транзакций;
- временной смены флагов;
- временной подмены `locale`, масок сигналов, параметров логирования;
- `scope exit`-логики.

Упрощенный пример:

```cpp
class RollbackFlag {
public:
    explicit RollbackFlag(bool& flag) noexcept : flag_(flag), old_(flag) {
        flag_ = true;
    }

    void commit() noexcept {
        active_ = false;
    }

    ~RollbackFlag() noexcept {
        if (active_) {
            flag_ = old_;
        }
    }

private:
    bool& flag_;
    bool old_;
    bool active_{true};
};
```

## RAII и exception safety guarantees

RAII - один из главных инструментов построения `exception-safe` кода.

Классические гарантии:

### Basic guarantee

После исключения программа остается в валидном состоянии, ресурсы не утекли.

### Strong guarantee

Операция либо завершилась успешно, либо состояние осталось как до операции.

### No-throw guarantee

Операция гарантированно не бросает исключений.

RAII сам по себе прежде всего обеспечивает:

- корректный `cleanup`;
- отсутствие утечек;
- упрощение `rollback`-подобных сценариев.

Например, если временный объект-guard откатывает изменение состояния в деструкторе, можно строить более сильные гарантии.

Важно: RAII сам по себе не гарантирует `strong guarantee` или `no-throw guarantee`. Он гарантирует корректное освобождение ресурсов и сильно упрощает построение более сильных гарантий.

## RAII как scope-bound resource management

Иногда RAII описывают как `scope-bound resource management`.

То есть ресурс живет ровно столько, сколько живет область видимости guard-а.

## Где RAII особенно полезен

### Системный C/C++ код

Работа с дескрипторами, сокетами, памятью, `lock`-ами.

### Многопоточный код

Управление `lock lifetime` через `lock_guard` / `unique_lock`.

### Код с исключениями

Чтобы не писать `cleanup` вручную по всем веткам.

### Сложные функции с несколькими точками выхода

RAII делает `early return` безопасным.

### Библиотечный код

Когда нужно формализовать владение ресурсом на уровне API.

## Признаки хорошего RAII-класса

Хороший RAII-тип обычно обладает такими свойствами:

- ясно выраженный `owner`;
- корректный инвариант после успешного конструктора;
- `destructor` безопасен и не бросает;
- `ownership semantics` очевидна;
- копирование либо запрещено, либо осмысленно;
- `move support` реализован корректно;
- `invalid` / `empty` / `moved-from state` корректен;
- ресурс освобождается ровно один раз;
- `API` делает неправильное использование трудным.

## Почему выброс из конструктора допустим

Конструктор может бросить исключение, потому что если ресурс не удалось получить, то объект не должен существовать в полусломанном состоянии.

Корректная философия такая:

- либо `constructor succeeds`, и объект валиден;
- либо `constructor throws`, и объект не создан.

Это намного лучше, чем создавать "мертвый" объект, у которого потом надо вызывать `is_valid()` перед каждым использованием.

## RAII и deterministic destruction vs garbage collection

RAII особенно силен тем, что `cleanup` детерминированный:

- известен точный момент разрушения объекта;
- значит, известен точный момент `release` ресурса.

В языках с `GC` память может быть освобождена позже, а ресурсы вроде `file handle` / `lock` / `socket` часто нельзя отпускать "когда-нибудь потом". Для них `deterministic cleanup` особенно важен.

## Типичные ошибки и анти-паттерны

### Ручной `open()` + ручной `close()`

Если жизненный цикл ресурса не оформлен типом, почти всегда со временем появится утечка или двойное освобождение.

### Два owner-а для одного ресурса

Если один и тот же raw-resource считают своим сразу два объекта, легко получить `double free` / `double close`.

### Псевдо-валидный объект после неудачной инициализации

Если объект можно создать в "сломленном" состоянии, пользователь API начинает тащить по коду лишние проверки.

### Слишком широкий `scope`

RAII работает лучше всего, когда `scope` владения минимален. Держать `lock` или `file handle` дольше, чем нужно, вредно даже если утечек нет.

### Явный вызов деструктора

Ручной вызов `obj.~T()` почти никогда не нужен для обычного RAII-кода и легко ломает модель владения.

## Короткие best practices

- выражать владение типом, а не договоренностью в комментарии;
- делать `owner`-тип максимально маленьким и с простым инвариантом;
- запрещать копирование там, где ресурс нельзя безопасно копировать;
- проектировать корректное `moved-from` состояние;
- держать `dtor` `noexcept`;
- по возможности использовать готовые RAII-типы из стандартной библиотеки;
- сокращать `scope` владения до минимально необходимого;
- если освобождение может вернуть значимую ошибку, предоставлять отдельный явный метод кроме `dtor`.

## Что стоит запомнить

RAII - это общая модель `scope-bound ownership`.

Сильная сторона RAII:

- ресурс либо принадлежит объекту, либо нет;
- освобождение выполняется автоматически;
- корректность завязана на тип и `scope`, а не на дисциплину программиста;
- код становится проще, безопаснее и легче для локального reasoning.
